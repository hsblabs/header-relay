Chrome拡張機能の比較

ヘッダー機能だけでいい、より狭いRequestlyの代替を探していますか？

Requestlyは30万人以上の開発者に使われているオープンソースのHTTPインターセプターで、ヘッダー変更、APIモック、URLリダイレクト、レスポンス上書き、スクリプト挿入を1つの拡張機能でカバーします。Header Relayはそのうちの一部、選択したレスポンスヘッダーを取得し、固定値または取得した値を一致するリクエストへ中継する機能のみを提供します。ヘッダーまわりしか使っていないなら、より狭いツールの方が権限も少なく、レビューする範囲も小さくて済みます。

実際に違うのはどこか。

安全性の指摘ではありません — Requestlyはオープンソースで広く使われています。これは範囲のトレードオフです: 何でもできる1つのツールか、1つのことだけをするツールか。

単一目的

Header Relayはヘッダーの取得と中継のみを行います。Requestlyのモック、リダイレクト、スクリプト挿入に相当する機能はありません — その分、設定ミスの余地もありません。

小さくレビューしやすいコード

ヘッダーのライフサイクルロジックは、ChromeのdeclarativeNetRequest APIとローカルストレージのみを使う1つのモジュールに集約されています。

取得した値を確認できる

取得したヘッダー値はデフォルトでマスクされ、確認が必要なときだけ明示的に表示またはコピーできます。

アカウント不要、クラウド同期なし

プロファイルと取得した値はローカルのChromeストレージに留まります。サインインするものは何もありません。

ヘッダー機能だけの利用なら、乗り換えは3ステップです。

Header Relayは他の拡張機能のルールをインポートしません — ヘッダーだけの構成なら再現はすぐ終わります。

01

プロファイルを作成する

ヘッダールールを適用していた対象オリジンを追加してください。

02

固定ヘッダーを移行する

APIキーやクライアントIDなどの固定値はFixed Headerとして設定します。

03

取得値を自動化する

セッショントークンやトレースIDなど、レスポンスから抽出していた値はCaptured Headerとして設定すれば自動的に再付与されます。

Requestlyからの乗り換えについてのよくある質問。

Requestlyは安全ではないのですか？

いいえ、このページはRequestlyに関する問題を報告するものではありません。Requestlyはオープンソースで広く使われています。ここでの比較は純粋に範囲の違いについてです — Header Relayはより狭い役割のみを担います。

何を諦めることになりますか？

APIモック、レスポンスボディの上書き、URLリダイレクト、スクリプト挿入です。Header Relayはこれらを行いません。これらの機能が必要な場合は、Requestlyの方が適しています。

両方同時に使えますか？

基本的には可能です。Header Relayは有効なプロファイルの対象オリジンに対するヘッダーのみを扱うため、両方の拡張機能が同じリクエストの同じヘッダーを対象にしない限り競合しません。
